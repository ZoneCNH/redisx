# ADR-20260602-003: Core gate 与发布前置检查

## 状态

Accepted

## 背景

`redisx` 同时承担标准源、Go 模板、generator、Harness 和 Evidence Runtime 职责。发布入口如果分散在脚本、Makefile 和人工步骤中，容易出现分数、证据、工作区状态和 release manifest 不一致的问题。

## 决策

采用 `make release-check`、`make release-final-check`、`make release-preflight VERSION=<version>` 和 `make score` 作为发布核心 gate：

- `make score` 通过 `goalcli score --min 9.8` 维护发布分数阈值。
- `make release-check` 生成并验证 release evidence、manifest hash 和 checksum。
- `make release-final-check` 在 `XLIB_CONTEXT=release_verify` 与 `GOWORK=off` 下执行最终发布验证，并要求 release evidence 处于 passed 且工作区干净。
- `make release-preflight VERSION=<version>` 在发布打 tag 前验证当前分支、远端同步状态、版本号、changelog、tag 冲突和 final check。

所有发布验证必须使用 `GOWORK=off`，避免本地 `go.work` 改写 module 解析。

## 影响

- `.agent/harness/harness.yaml` 和 `cmd/goalcli` 是 gate 契约的机器可执行入口。
- `docs/release.md` 必须描述 release check、final check、preflight、tag 与 evidence 操作顺序。
- 发布完成声明必须引用最新 gate 输出和 release evidence，不得只依赖人工审查或历史日志。
- 任何新增发布 gate 都必须同步 Makefile、Harness、文档和 traceability matrix。
